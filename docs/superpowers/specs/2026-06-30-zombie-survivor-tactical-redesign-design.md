# Zombie Survivor 战术伪 3D 改造设计

## 1. 目标与成功标准

在不更换 TypeScript、Canvas 2D、自研 ECS 和现有确定性模拟的前提下，把当前版本改造成更清晰、更有冲击力的“战术末日 × 预渲染伪 3D”生存 Roguelite。

成功标准：

1. 1280×720 下，玩家、近身敌人、经验、金币和危险攻击可在一眼内区分。
2. 新玩家能在开始后 30 秒内完成“移动、击杀、拾取、首次升级”反馈闭环。
3. HUD 明确展示阶段目标、时间、生命、经验、金币、技能和武器进度，不遮挡主要战场。
4. 商店、升级和结算界面使用统一图片资产，不再以 Emoji 作为正式图标。
5. 玩家和敌人具备可感知的移动、攻击、受击和死亡姿态；伪 3D 深度不依赖真正 3D 引擎。
6. 满屏高敌人数量时不引入逐实体昂贵滤镜，现有测试、构建和确定性模拟全部通过。

视觉目标见 [`selected-direction.png`](../../design/2026-06-30-tactical-redesign/selected-direction.png)。

## 2. 设计原则

### 战场优先

至少 75% 画面留给战斗。信息固定在边缘，升级、商店和结算才使用居中覆盖层。

### 四色语言

- 青色：经验、升级、成长和冷却就绪。
- 琥珀：玩家火力、金币和可购买动作。
- 珊瑚红：生命损失、危险单位和威胁提升。
- 洋红：Boss、精英能力和不可忽略的阶段警报。

### 深度来自层次而非引擎

通过脚底锚点、椭圆投影、按 Y 排序、人物垂直偏移、环境暗角、局部光晕和前后景密度构建伪 3D。不开启 Three.js，不替换 ECS，不修改碰撞坐标系。

### 先读清，再变爽

每种特效只承担一个信息任务。命中、暴击、击杀、拾取、受伤、升级和阶段切换分别使用稳定的形状、颜色和声音，不用同一种闪光覆盖全部事件。

## 3. 玩家体验

### 开始界面

- 使用选定方案中的角色主视觉、任务名称和三项核心提示。
- 显示“移动 / 自动开火 / 拾取升级”三个短说明，不再用三行长句解释全部系统。
- 保留一个主按钮和 Space 快捷键。

### 首局 30 秒

- 前 12 秒减少刷怪压力，避免玩家尚未移动就被包围。
- 前 30 秒扩大基础拾取范围，帮助玩家自然理解经验和金币回收。
- 首次拾取经验、首次受伤和首次可升级时，各显示一次短提示；之后本局不重复。
- 不增加独立教程关，不阻断战斗。

### 阶段节奏

- 阶段切换时显示 1.5 秒边缘横幅和威胁等级，不暂停游戏。
- 进入新阶段给予少量补给：回复一小部分生命并获得固定金币，形成可感知的节奏节点。
- Boss 前 15 秒沿用现有警告逻辑，但增加顶部倒计时和洋红危险色。

### Build 决策

- 升级卡显示类型、当前等级、变化值和协同标签。
- 商店按“生存 / 火力 / 战术”显示，并明确金币差额、已持有状态和快捷键。
- 保留随时按 B 打开商店的连续玩法，不改成波间强制商店，避免重写当前节奏和状态机。

### 结算

- 展示生存时间、击杀、到达阶段、主要武器、金币和死亡原因。
- 给出一个下一局目标，例如“到达第 2 阶段”或“升级一把武器到 Lv.3”。
- 本轮不新增完整跨局货币树；长期元进度留作后续独立需求，避免一次改造扩大为存档系统重写。

## 4. UI 结构

### HUD

- 顶部：6px 经验条；中央阶段、时间、威胁；右上金币和商店按钮。
- 左下：宽 320px 的生命/护盾组合条，生命数字至少 16px。
- 下中：消耗品和主动技能槽，统一 52px 尺寸，冷却用遮罩与数字双重表达。
- 右下：当前主武器、等级与升级进度；其他武器收成紧凑列表。
- Boss 出现时，Boss 血条占顶部中央第二层，不与阶段信息重叠。

### 覆盖层

- 使用统一的深黑青面板、1px 低对比描边、有限阴影和更高正文对比度。
- 升级三选一保持单行三卡；商店使用分类标签和最多四列；结算使用两列摘要。
- 所有可点击卡片提供 hover、focus-visible 和不可购买状态。

### 响应式边界

- 以 1280×720 为主验收尺寸。
- 宽度低于 900px 时隐藏次要武器列表并压缩顶部文案，不缩小关键生命数字和技能槽。
- 本轮不新增移动端触控摇杆。

## 5. 渲染、动作与特效

### 伪 3D 渲染

- 地面增加低成本暗色渐变、碎屑遮罩和摄像机相对视差偏移。
- 活体单位按脚底 Y 坐标排序后绘制，尸体和血迹仍在底层。
- 投影根据单位速度和受击状态轻微拉伸；Boss 投影更大、更软。
- 玩家周围使用低透明暖色局部光，环境保持冷色，增强主体分离。

### 人物动作

- 保留现有 bob、squash、rock，并提高玩家可读幅度。
- 玩家依据移动速度增加前倾；射击冷却刚重置时产生短促后坐；受击时产生后仰和闪白。
- 敌人靠近攻击距离时产生短促前扑；重型敌人降低步频、提高身体摆幅；Boss 使用更慢更重的呼吸和落脚节奏。
- 死亡继续使用现有倾倒尸体，但增加短时压扁、暗化和尘土冲击。

### 特效预算

- 普通命中：4 个火花 + 小伤害数字。
- 暴击：更大火花、较粗数字、短命中停顿。
- 击杀：敌人色爆裂、血迹、金币/经验吸附线。
- 玩家受伤：珊瑚红边缘闪光、9px 级震屏和短无敌闪烁。
- 不对每个敌人使用 `shadowBlur`、滤镜或离屏 Canvas；高频效果继续使用对象池。

## 6. 图片资产

- 保留现有玩家、敌人、武器和地面图片作为第一轮资产基础。
- 新增 8 个统一风格的装备图标：磁能吸附、高爆手雷、急救包、护盾发生器、疾风战靴、狂暴药剂、金币倍增、死亡之舞。
- 装备图标为 96×96 PNG，暗色战术物件、琥珀/青色功能光、无文字、透明背景。
- 更新 manifest 和资产说明，商店和 HUD 通过 `iconKey` 使用图片，不再直接渲染 Emoji。

## 7. 代码边界与数据流

### 保留边界

- ECS、碰撞、武器和敌人定义保持数据驱动。
- 视觉状态不进入确定性模拟，除非它直接表达阶段奖励等真实玩法结果。
- DOM UI 仍由 `UI` 类负责，Canvas 仍由 `Game` 和 `Renderer` 负责。

### 新增/调整单元

- `src/ui/ui.ts`：HUD 结构、覆盖层、阶段横幅和结算摘要。
- `src/game.ts`：渲染排序、视觉姿态、阶段变化检测和 UI 数据组装。
- `src/render/renderer.ts` 与 `src/render/canvas2d.ts`：屏幕空间色调、光晕或渐变所需的最小绘制能力。
- `src/data/balance.ts`：首局节奏和阶段补给的纯数据配置。
- `src/data/equipment.ts`：Emoji 改为图片 `iconKey`。
- `public/assets/`：新增装备图标并更新 manifest/说明。

数据流：模拟系统更新世界状态 → `Game` 读取世界与阶段状态 → Canvas 绘制战场 → `UI.updateHud` 更新边缘信息。阶段变化产生一次真实补给，再触发一次纯视觉横幅。

## 8. 错误处理

- 图片加载失败时继续使用现有几何占位，不阻断游戏。
- 新图标缺失时卡片显示物品名称，不回退为 Emoji。
- 阶段奖励只在阶段索引真实递增时触发一次，暂停、商店和升级状态不会重复触发。
- 新手提示只保存在本局状态，不写入长期存档。

## 9. 验证

### 自动验证

- `npm test`
- `npm run build`
- 新增针对早期刷怪节奏、阶段奖励只触发一次和装备图标数据完整性的测试。

### 浏览器验收

- 1280×720 下检查开始、前 30 秒、首次升级、商店、阶段切换、受伤、阵亡和 Boss 警告。
- 对比当前截图与选定视觉目标，确认玩家尺寸、HUD 层级、颜色语言和伪 3D 深度明显改善。
- 检查键盘操作、focus-visible、文本对比、缩放和 900px 宽度回流。
- 记录控制台错误与关键截图；性能上确认常规战斗无明显卡顿。

## 10. 非目标

- 不迁移到 Phaser、Three.js 或 React。
- 不实现真正 3D 模型、骨骼动画或动态光照系统。
- 不新增联网、账号、排行榜、移动端触控或完整元进度树。
- 不重做全部现有角色和敌人原画。
